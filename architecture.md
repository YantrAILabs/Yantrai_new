# Architecture

## Overview

YantrAI is a local-first marketing web app with a static frontend and a lightweight Python backend. The frontend remains plain HTML, CSS, and JavaScript, while backend routes are used for operational actions such as demo-request email delivery.

## Components

- `index.html`: Main page markup and inline SVG sections.
- `styles.css`: Site styling and responsive layout.
- `script.js`: Browser interactions, including reveal behavior, tabs, mobile burger menu, and demo form submission.
- `main.py`: Flask app serving static files and handling API routes.
- `requirements.txt`: Runtime dependencies (`Flask`, `gunicorn`) for App Engine Python deployment.
- `assets/`: Static images used by the site.
- `app.yaml`: Google App Engine deployment configuration with static handlers + dynamic Python app handler.

## Hosting Flow

The app serves static files and handles API calls:

```text
Browser request to appspot.com or custom domain
  -> App Engine handler
    -> static handler for html/css/js/assets
    -> Flask route fallback for dynamic endpoints
       -> /api/book-demo -> SMTP email send
```

`/api/book-demo` accepts JSON from the form (`name`, `company`, `role`, `mobile`, `email`, `challenge`), validates required fields, and sends a formatted email to `rohit@yantrailabs.com` (default recipient), using environment-configured SMTP settings and a Secret Manager-backed SMTP password.

## Deployment

Deploy with:

```bash
gcloud config set project yantraivisionos
gcloud app create --region=asia-south1
gcloud app deploy
```

The deployment uses the active Google Cloud project selected in the local `gcloud` CLI. App Engine app creation is a one-time setup step; the region cannot be changed after creation.

## API Contract

### `POST /api/book-demo`

- Request body: JSON
  - Required: `name`, `company`, `role`, `mobile`
  - Optional: `email`, `challenge`
- Success response: `200 { "ok": true }`
- Validation failure: `400 { "ok": false, "error": "Missing required fields" }`
- Email configuration/runtime failure: `5xx` with `{ "ok": false, "error": "..." }`

## Decisions

- App Engine was chosen because Firebase project initialization failed after the Google Cloud project was created.
- A lightweight Flask backend was introduced to support real form delivery without adding a separate service.
- `app.yaml` still exposes only common public file types as static assets while routing all other paths to Python for API handling.
- SMTP password is not committed in plaintext; runtime reads `SMTP_PASS_SECRET` from Secret Manager.
- `automatic_scaling.max_instances` is set to `2` to make App Engine's scaling limit explicit for this low-traffic static site.

## Constraints

- Deployment requires the Google Cloud CLI.
- First deploys in new projects can require Storage Admin on the App Engine default service account so Cloud Build can use the staging bucket.
- App Engine's buildpack deployment can use Artifact Registry through `asia.gcr.io`; new projects may need `roles/artifactregistry.createOnPushWriter` on the App Engine default service account.
- SMTP host/user/from/to values are configured in `env_variables`, while SMTP password is fetched from Secret Manager via `SMTP_PASS_SECRET`; App Engine service account needs `roles/secretmanager.secretAccessor`.
- Custom domains are mapped in App Engine and then pointed from GoDaddy DNS using the records generated by Google Cloud.
- App Engine issues managed SSL certificates for mapped custom domains, but activation can lag behind DNS propagation.
- The top navigation and agent CTA links now route only to in-page anchors to avoid dead-link 404s.
